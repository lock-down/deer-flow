DeerFlow 部署与状态管理
Deployment & State Management
1. 本地开发部署
启动流程
通过 Makefile 统一管理所有服务
make check # 检查系统依赖 make install # 安装前后端依赖 make config # 生成 config.yaml (首次) make config-upgrade # 合并配置新字段 make setup # 交互式设置向导 make doctor # 检查配置和系统 make dev # 启动所有服务 (标准模式) make dev-pro # 启动所有服务 (网关模式)
→ Nginx :2026 (统一入口)
→ Frontend :3000 (Next.js)
→ Gateway :8001 (FastAPI)
→ LangGraph :2024 (Agent)
停止与清理
make stop # 停止所有服务 make clean # 停止 + 清理临时文件
环境要求
Node.js >=22
pnpm 10.26.2+
Python >=3.12
uv (Python 包管理)
nginx
运行时模式 (Runtime Modes)
Standard 标准模式 make dev
4 进程: Nginx + Frontend + Gateway + LangGraph
Agent 运行在独立 LangGraph Server(:2024)
Nginx 路由 /api/langgraph/* → LangGraph
Gateway 网关模式 make dev-pro
3 进程: Nginx + Frontend + Gateway(embedded)
Agent 运行时嵌入 Gateway via RunManager + StreamBridge
无需独立 LangGraph Server

3. ThreadState 状态结构
ThreadState (extends AgentState)
class ThreadState(AgentState): messages: list[Message] # 对话消息历史 sandbox: SandboxState | None # 沙箱执行状态 thread_data: ThreadDataState | None # 线程元数据 title: str | None # 线程标题 artifacts: list[str] # 工件列表 (归并去重) todos: list | None # 任务列表 (plan_mode) uploaded_files: list[dict] # 已上传文件 viewed_images: dict # 已查看图片 (归并)
State Flow:
Input
→
AgentNode
→
ToolsNode
→
Middleware
→
Output
重要 归并策略: artifacts 和 viewed_images 使用特殊归并去重逻辑，避免重复

5. Sandbox 虚拟路径映射
虚拟路径 → 实际路径
/mnt/user-data/ → .deer-flow/threads/{thread_id}/user-data/
/workspace/ → user-data/workspace/
/uploads/ → user-data/uploads/
/outputs/ → user-data/outputs/
Agent 在沙箱中访问文件时使用虚拟路径，系统自动映射到实际线程目录
Sandbox 类型
Local 本地沙箱 (开发)
Docker Docker 沙箱 (隔离)
K8s Kubernetes 沙箱 (生产)
Sandbox Tools
bash → 命令执行
ls → 文件列表
read_file → 文件读取
write_file → 文件写入
str_replace → 文件替换
glob → 文件模式搜索
grep → 内容正则搜索
# config.yaml sandbox: use: deerflow.sandbox.local:LocalSandboxProvider allow_host_bash: false
安全 security.py 门控 bash 执行 · file_operation_lock.py 使用 WeakValueDictionary 防止内存泄漏
